初始化 org.apache.pulsar.client.impl.PartitionedProducerImpl

  1. 设置状态为 Uninitialized
  2. 初始化router
  3. 统计时间间隔设置
    stats = client.getConfiguration().getStatsIntervalSeconds() > 0 ? new ProducerStatsRecorderImpl() : null;
  4. 设置最大等待发送消息
    默认是1000和50000/分区数的最小值
    Math.min(conf.getMaxPendingMessages(),conf.getMaxPendingMessagesAcrossPartitions() / numPartitions);
  5. 启动分区生产者
    start();
  6. autoUpdatePartitions 配置为true(默认为true) 启动跟踪和自动订阅分区增加
    1. topicsPartitionChangedListener = new TopicsPartitionChangedListener();
    2. autoUpdatePartitionsIntervalSeconds 自动更新分区数量时间间隔，默认60s
      1. 自动扩容只支持分区的topic,且所有topic不应该含有 “-partition-“,因为这是用这个判断的，不知道topic创建的时候会不会有判定
      2. 扩容完毕后会重新设置一次topicMetadata
      3. 如果扩容失败则会回退
    3. 不支持缩容

启动 org.apache.pulsar.client.impl.PartitionedProducerImpl

  1. 如果topicName中含有 “-partition-“ 那么不会自动扩展缀，直接使用topicName
  2. 最终转到初始化org.apache.pulsar.client.impl.ProducerImpl，即无分区的生产者
  3. 如果所有分区生产者都创建成功，那么设置状态为Ready,否则关闭所有已经创建好的生产者
    client.cleanupProducer(this);

初始化org.apache.pulsar.client.impl.ProducerImpl

  1. 设置状态为 Uninitialized

  2. 初始化生产者id和名称

    1. id 为自增
    2. 名称如果不设置就是没有

  3. 构建待发送消息队列数据结构为 java.util.ArrayDeque

  4. 构建信号量 公平信号量 java.util.concurrent.Semaphore.FairSync

  5. 根据配置 compressionType 设置压缩模式，默认NONE

  6. 设置发送序列Id，如果不设置，默认是-1

    1. lastSequenceIdPublished -1
    2. lastSequenceIdPushed -1
    3. msgIdGenerator 0

  7. 如果启用加密

    1. 默认使用 org.apache.pulsar.client.impl.crypto.MessageCryptoBc#MessageCryptoBc
    2. 且4小时一次换取公钥？

  8. 默认发送超时为 30s

  9. 开启批量消息

    1. 默认不分组
      1. DEFAULT 不分组 org.apache.pulsar.client.impl.DefaultBatcherBuilder
      2. KEY_BASED 按key分组 org.apache.pulsar.client.impl.KeyBasedBatcherBuilder

  10. 初始化监控指标 statsIntervalSeconds > 0

    1. stats = new ProducerStatsRecorderImpl(client, conf, this);

  11. 使用 conf.getProperties() 给metadata设置值

  12. 初始化链接处理器 TODO

  13. 抓取连接 grabCnx();

    cnx connection
    this.connectionHandler.grabCnx();

  14. 判断链接状态

    1. 如果不为空则直接返回。
    2. 如果状态是无效的则直接返回
    3. Closing
    4. Closed
    5. Failed
    6. ProducerFenced
    7. Terminated

  15. 创建链接

    1. lookup.getBroker(topicName) 获取broker地址（轮询随机的broker地址)
      1. topic具体由哪些broker管理不确定？ TODO
    2. 按逻辑地址为key，再以随机产生的每个broker可以创建的链接数以下的值为key创建链接
    3. 直接使用物理地址进行链接
    4. 链接创建后
      1. 增加关闭监听
      2. 如果逻辑地址和物理地址不一致，那么要标记出来 this.proxyToTargetBrokerAddress

  16. 执行创建完毕后的操作
    org.apache.pulsar.client.impl.ConnectionHandler.Connection#connectionOpened
    我们目前关注的是生产者
    org.apache.pulsar.client.impl.ProducerImpl#connectionOpened

org.apache.pulsar.client.impl.ProducerImpl#connectionOpened

  1. connectionHandler.setClientCnx(cnx);

    1. 如果连接中断，那么connectionHandler会自动重连

  2. cnx.registerProducer(producerId, this);

    1. 向链接注册生产者

  3. 校验一下schema

  4. cnx.sendRequestWithId()

    1. 发送创建producer的请求

      1. Commands.newProducer() TODO 猜测是命令模式，待看
      2. 如果创建失败
        1. 移除生产者
        2. 如果生产者已处于Closing|Closed则关闭链接频道并返回
        3. 如果topic不存在则执行关闭生产者org.apache.pulsar.client.impl.ProducerImpl#closeAsync
        4. 如果backlog已经存在但是配额不足导致的异常则清除所有待发送的消息并抛出异常 ProducerBlockedQuotaExceededException
          1. 如果链接不存在则 TODO 这里有很多操作啊
            1.
          2. 如果链接存在则 TODO 不明白
        5. 如果backlog已不足以创建生产者 ProducerBlockedQuotaExceededError
        6. … TODO 异常处理

    2. 获取producer的一些信息

      1. schemaVersion
      2. lastSequenceId

    3. resetBackoff(); TODO

    4. 如果是在该服务器首次建立生产者，那么设置

      1. lastSequenceIdPublished = 服务器返回的lastSequenceId
      2. msgIdGenerator = lastSequenceId + 1

    5. if (!producerCreatedFuture.isDone() && isBatchMessagingEnabled())

      1. 暂时不能理解出现这种情况的原因 TODO
        1. 看着像是如果是partition情况，会出现生产者还未完全建立的情况
      2. 启动定时任务批量发送消息
        1. batchingMaxPublishDelayMicros 默认1ms
        2. batchMessageAndSend();

    6. resendMessages(cnx);

      1. 如果待发送消息为空则标记生产者初始化完成
      2. 如果不为空则启动数据恢复操作(重新发送未发送的消息)
